Questions fréquentes (FAQ)

1. Comment fonctionne l’analyse de sécurité ?

L’analyse de sécurité utilise une API serverless qui récupère les en‑têtes (headers) du site analysé. Ces informations permettent de vérifier des protections importantes comme : HTTPS, CSP, HSTS, X‑Frame‑Options ou encore les cookies sécurisés.

2. Pourquoi l’analyse de sécurité ne fonctionne pas avec du HTML collé ?

Les informations de sécurité proviennent du serveur, pas du code HTML. Elles ne peuvent être obtenues qu’en analysant une URL réelle.

3. Pourquoi certaines URL ne peuvent pas être analysées ?

Même avec l’API serverless (qui contourne le CORS), certains sites bloquent les requêtes externes. Cela arrive lorsqu’ils utilisent des protections comme :

  • Cloudflare (mode anti‑bot)
  • des firewalls applicatifs
  • des restrictions d’IP
  • des systèmes anti‑scraping

Dans ce cas, l’analyse renvoie : “Impossible d’analyser ce site (bloqué ou invalide).”

4. L’API serverless supprime‑t‑elle le problème CORS ?

Oui. Grâce à l’API, l’analyse passe par un serveur intermédiaire et non par votre navigateur. Le CORS ne s’applique donc plus. Seuls les sites protégés par des systèmes anti‑bot peuvent encore refuser l’analyse.

5. Comment analyser un site si l’URL ne fonctionne pas ?

Vous pouvez toujours analyser l’accessibilité du site en :

  1. Ouvrant la page dans votre navigateur
  2. Faisant CTRL + U (ou clic droit → Afficher le code source)
  3. Copiant tout le code HTML
  4. Le collant dans la zone prévue sur la page d’accueil

6. Comment fonctionne l’analyse d’accessibilité ?

L’analyse d’accessibilité examine directement le code HTML fourni. Elle vérifie notamment :

  • la présence d’attributs alt sur les images
  • la présence de label pour les champs de formulaire
  • la structure des titres (h1, h2, h3)

Un score et un résumé automatique sont ensuite générés.

7. Quelle est la différence entre sécurité et accessibilité ?

La sécurité protège les données et empêche les attaques. L’accessibilité garantit que le site peut être utilisé par tout le monde, y compris les personnes en situation de handicap. Les deux sont essentiels, mais ils analysent des aspects différents.

8. Pourquoi mon site apparaît “non sécurisé” alors qu’il fonctionne ?

Un site peut fonctionner normalement tout en ayant des failles de sécurité. Par exemple, il peut manquer de protections comme HSTS, CSP ou des cookies sécurisés. Cela n’empêche pas l’accès au site, mais augmente les risques pour les utilisateurs.

9. Est‑ce que ce scanner détecte les virus ou les malwares ?

Non. Ce scanner analyse uniquement :

  • les bonnes pratiques de sécurité web
  • les bonnes pratiques d’accessibilité

Il ne remplace pas un antivirus ou un outil de détection de malware.

10. Est‑ce que l’analyse est fiable ?

Oui, pour les points analysés. Le scanner se base sur des bonnes pratiques reconnues (HTTPS, CSP, alt, labels…). Cependant, il ne peut pas analyser :

  • le code JavaScript interne
  • les bases de données
  • les failles avancées côté serveur

C’est un outil de sensibilisation, pas un audit complet.

11. Mes données sont‑elles enregistrées ?

Non. Aucune URL, aucun code HTML et aucun résultat n’est stocké sur un serveur. Tout est géré localement dans votre navigateur via localStorage.
